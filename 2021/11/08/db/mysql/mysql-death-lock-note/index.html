mysql死锁记录

死锁发生的场景

模拟死锁

  • 模拟订单数3千笔，然后同一时刻把消息发送至MQ，消费端分为C端消费和B端消费
    • 1、TestNG录制了从创建客户到下订单的过程
    • 2、然后把订单号摘选出来，利用模拟接口把所有的数据发送至RabbitMQ
  • 其中C端消费会报获取锁等待超时错误
  • 其中B端消费会报数据库死锁

程序报错说明

  • 其中B端的费用明细新增会产生死锁
    • 报错现象为：新增费用明细会报数据库死锁
    • 原因为：新增费用明细和更新客户状态费用明细表的索引字段并发导致数据库死锁
    • 解决方式，高并发阶段停止更新客户状态的定时任务
  • 其中C端的回调，更新订单状态并发量大的情况会出现获取锁等待超时的情况
    • 原因为并发量超过update的限制
    • 解决方式，rabbitmq消费端限流

  • 解决死锁之路（终结篇） - 再见死锁

1、执行show engine innodb status;

2、拿到日志，日志中列出了死锁发生的时间，以及导致死锁的事务信息（只显示两个事务，如果由多个事务导致的死锁也只显示两个），并显示出每个事务正在执行的 SQL 语句、等待的锁以及持有的锁信息等。下面我们就来研究下这份死锁日志，看看从这份死锁日志中能不能发现死锁的原因？。

3、第一条语句为新增费用明细的语句

4、第二条语句为更新客户状态，同时运用了费用明细的索引字段，猜测该原因造成死锁

5、遂再造相同的数据，然后把定时关闭，发现并未造成数据库的死锁

6、扩展：如何在本地用SQL模拟出以上的场景困扰到了我。死锁还有挺多可以挖掘的。
